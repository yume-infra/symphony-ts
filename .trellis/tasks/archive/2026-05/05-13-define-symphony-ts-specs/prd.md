# Define Symphony-ts project specs

## Goal

Replace the initial fullstack Trellis spec template with a Symphony-ts-specific spec library that
future Codex `/goal` runs can load before implementing the runtime.

The first version should capture the current project decisions without trying to finish every future
implementation detail. A later task will review and revise these specs before the long implementation
loop starts.

## Confirmed Facts

- The repository is a TypeScript ESM package generated by `create-yume`.
- The public command shape should stay minimal: `symphony-ts [workflow-path]`.
- The product is a long-running orchestration service, not a traditional frontend/backend app.
- `SPEC.md` is the reference blueprint and terminology source.
- `SPEC.md` should be treated as a conformance baseline with explicit project deviations, not as a
  document to blindly copy.
- Effect is the core application architecture layer.
- The project intentionally uses the experimental `@effect/tsgo` toolchain for Effect diagnostics.
- AI/coding-agent infrastructure is part of the product surface and must be planned as a first-class
  concern.
- OpenAI Symphony's `.codex/` setup is useful reference material, but should not be copied directly.

## Requirements

- Replace the existing `backend/` and `frontend/` Trellis spec layers with project-specific layers:
  - `symphony/`
  - `runtime-orchestration/`
  - `external-integrations/`
  - `typescript-effect/`
  - `testing-conformance/`
  - `quality-operations/`
- Keep `.trellis/spec/guides/` unless a guide clearly conflicts with Symphony-ts.
- Each layer must include an `index.md` with:
  - scope
  - pre-development checklist
  - quality check
  - links to detailed guideline files
- Specs must distinguish:
  - baseline rules from `SPEC.md`
  - project-specific decisions
  - explicitly allowed deviations
  - unresolved decisions for the next planning task
- The specs must be useful to a Codex CLI `/goal` run without needing this chat transcript.
- The task must also record an implementation context plan for future `/goal` usage.

## Acceptance Criteria

- [x] `.trellis/spec/backend/` and `.trellis/spec/frontend/` are removed.
- [x] New project-specific spec layers exist with concrete, non-placeholder content.
- [x] Effect usage rules reference the current project toolchain and `@effect/tsgo` setup.
- [x] Runtime specs cover workflow loading, config, orchestrator state, workspaces, retries, and
      reconciliation at a level suitable for implementation.
- [x] Integration specs cover Linear, Codex app-server, and optional client tools.
- [x] Testing specs map `SPEC.md` sections 17 and 18 into project validation expectations.
- [x] Quality/operations specs cover logs, safety invariants, AI infrastructure, and `/goal` usage.
- [x] The task has `design.md` and `implement.md`.
- [x] Context manifests point future implement/check work at the new spec entry points.
- [x] `python3 ./.trellis/scripts/get_context.py --mode packages` reflects the new spec layers.
- [x] Project verification still passes.

## Out Of Scope

- Implementing the Symphony runtime.
- Building Linear, Codex app-server, workspace, or orchestrator modules.
- Creating final `.codex/skills` automation.
- Vendoring the Effect source tree.
- Finalizing product deviations that still require user discussion.
